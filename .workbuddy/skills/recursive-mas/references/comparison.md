## 1. 架构对比

### 传统文本中介 MAS

```
Agent₁ → [decode text] → [encode text] → Agent₂ → [decode text] → ...
```

每个 Agent：
1. 接收文本输入
2. 生成文本输出
3. 下一个 Agent 重新编码文本

### RecursiveMAS

```
Agent₁ → [ℛ_out: latent vector] → Agent₂ → [ℛ_out: latent vector] → ...
```

每个 Agent：
1. 接收隐向量 + 自身指令嵌入
2. 在隐空间自回归生成 latent thoughts
3. 通过 ℛ_out 传递给下一个 Agent

### 核心区别

| | 传统 MAS | RecursiveMAS |
|--|---------|-------------|
| 信息载体 | 离散文本 | 连续隐向量 |
| 信息瓶颈 | 词汇表大小 | 隐维度（远大于词表选择） |
| 可组合性 | 仅文本拼接 | 隐空间拼接 + 投影 |
| 递归能力 | 需完整编解码 | 直接隐空间循环 |

## 2. 通信机制对比

### 文本通信的信息损失

1. **量化损失**：连续隐状态 → 离散 token，信息压缩
2. **语义偏移**：同一文本可被不同模型解读为不同语义
3. **格式依赖**：需要精心设计 prompt 格式
4. **长度限制**：文本越长，后续 Agent 上下文越拥挤

### 隐空间通信的优势

1. **无损传递**：连续向量保留完整语义信息
2. **分布对齐**：RecursiveLink 学习跨模型的语义映射
3. **维度自由**：不同 dₕ 的模型可通过 W₃ 无缝连接
4. **高效拼接**：隐向量可直接 concatenation，无需格式化

### 隐空间通信的挑战

1. **可解释性**：隐向量不易人工审查
2. **对齐需求**：需训练 RecursiveLink 实现跨模型语义对齐
3. **初始不稳定**：未训练的 RecursiveLink 输出可能无意义
4. **调试难度**：无法直接查看中间通信内容

## 3. 训练范式对比

### 传统 MAS 训练方式

| 方式 | 描述 | 优缺点 |
|------|------|--------|
| Prompt 优化 | 迭代改进共享上下文 | ✅ 简单 ❌ 效果有限 |
| 独立微调 | 各 Agent 独立 SFT/LoRA | ✅ 各自提升 ❌ 无系统协同 |
| 全系统微调 | 端到端训练所有参数 | ✅ 最佳效果 ❌ 计算成本极高 |
| TextGrad | 文本梯度优化 | ✅ 可解释 ❌ 梯度消失问题 |

### RecursiveMAS 训练方式

**仅训练 RecursiveLink**：
- 可训练参数 < 2% 总参数
- Inner-Loop 预热 + Outer-Loop 系统优化
- 梯度沿递归路径稳定传播
- 系统级端到端协同优化

### 训练效率对比

| 方法 | 可训练参数 | GPU 时间 | 效果提升 |
|------|-----------|---------|---------|
| Full FT | 100% | ~100h | 基线 |
| LoRA | ~5% | ~20h | 接近 Full FT |
| Prompt 优化 | 0% | ~1h | 最小 |
| RecursiveLink | ~1-2% | ~5-8h | +8.3% 平均 |

## 4. 性能对比

### 基准测试结果（9 项基准平均值）

| 方法 | 准确率提升 | 推理速度 | Token 节省 |
|------|-----------|---------|-----------|
| 单 Agent (FT/LoRA) | baseline | 1× | 0% |
| 单 Agent (LoopLM) | +3.1% | 0.8× | -15% (更多 token) |
| 文本递归 MAS | +2.7% | 0.6× | -40% (更多 token) |
| TextGrad | +4.5% | 0.7× | -10% |
| MoA (Mixture-of-Agents) | +5.8% | 0.5× | -60% |
| **RecursiveMAS** | **+8.3%** | **1.2-2.4×** | **34.6-75.6%** |

### 分领域表现

| 领域 | 基准 | 单 Agent | RecursiveMAS | 提升 |
|------|------|---------|-------------|------|
| 数学 | MATH500 | 62.4% | 71.2% | +8.8% |
| 数学 | AIME2026 | 26.7% | 33.3% | +6.6% |
| 科学 | GPQA-Diamond | 48.1% | 53.9% | +5.8% |
| 医学 | MedQA | 72.3% | 81.5% | +9.2% |
| 代码 | LiveCodeBench-v6 | 41.2% | 49.8% | +8.6% |
| 代码 | MBPP Plus | 58.9% | 67.1% | +8.2% |
| 搜索QA | HotpotQA | 55.6% | 63.8% | +8.2% |
| 搜索QA | Bamboogle | 61.3% | 70.4% | +9.1% |

### Token 效率分析

| 协作模式 | 递归轮次 | Token 节省 |
|---------|---------|-----------|
| Sequential (Light) | 2 | 75.6% |
| Sequential (Scaled) | 2 | 68.3% |
| Mixture | 2 | 34.6% |
| Distillation | 2 | 52.1% |
| Deliberation | 2 | 41.8% |

Sequential 模式节省最多，因为传统链式 MAS 每步都需完整编解码。

## 5. 适用场景分析

### RecursiveMAS 最适合的场景

1. **多轮推理任务**：数学、逻辑、科学推理
   - 递归精炼天然适合逐步深化的推理过程

2. **多领域综合任务**：需要不同专长的 Agent 协作
   - 隐空间传递避免了文本中间结果的失真

3. **Token 预算受限场景**：边缘部署、API 调用计费
   - 34.6%-75.6% 的 Token 节省直接降低成本

4. **延迟敏感场景**：实时对话、在线推理
   - 1.2×-2.4× 推理加速

### 传统 MAS 仍然适合的场景

1. **需要人类可审查的中间过程**
   - 隐空间传递无法直接展示中间推理步骤

2. **即插即用的 Agent 组合**
   - RecursiveMAS 需要训练 RecursiveLink

3. **已有完善的 prompt 工程流程**
   - 文本通信与现有工具链（LangChain 等）兼容性更好

4. **Agent 来源不可信**
   - 隐空间传递比文本更难做安全审计

## 6. 与相关工作的关系

### 递归语言模型 (RLM) 系列

```
LoopLM (单模型递归)
    ↓ 扩展到多智能体
RecursiveMAS (多模型递归)
```

- LoopLM：同一模型反复应用，增加推理深度
- RecursiveMAS：异构模型递归协作，增加推理广度+深度

### ReDel (EMNLP 2024 Demo)

- ReDel 提供 LLM 递归多智能体的工具包
- 支持 Agent 灵活委派和组织结构
- 但 Agent 间仍通过文本通信
- RecursiveMAS 可视为 ReDel 的隐空间升级版

### ROMA (Recursive-Open-Meta-Agent)

- 层次化递归 Meta-Agent 框架
- 专注于 Agent 结构的自动发现和优化
- 与 RecursiveMAS 互补：ROMA 优化结构，RecursiveMAS 优化通信

### Mixture-of-Agents (MoA)

- 多 Agent 并行聚合
- 缺乏递归精炼机制
- RecursiveMAS 的 Mixture 模式可视为递归增强版 MoA

### TextGrad

- 通过文本"梯度"优化 Agent 输出
- 受限于文本介质的梯度消失
- RecursiveMAS 的理论分析（定理 4.1）解释了其局限性
